Guia de Sobrevivência e Migração: Web2py para Django com Cursor

1. Planos e Limites do Cursor AI

  • Plano Hobby (Grátis):
    • 50 consultas Premium (Claude 3.5 Sonnet / GPT-4o) por mês.
    • 2.000 conclusões de código (Cursor Tab).
    • Bloqueio rígido após o fim dos créditos (não há modo lento).
    • Risco: Criar várias contas pode gerar banimento por Hardware ID.
  • Plano Pro (US$ 20/mês):
    • 500 consultas Premium "Fast".
    • Consultas ilimitadas no modo "Slow" (após gastar as 500).
    • Uso ilimitado do Cursor Tab.
    • Modo de Privacidade (dados não são usados para treino).
  • API Keys Próprias:
    • Paga direto para OpenAI/Anthropic.
    • Sem limite de mensagens, mas custo pode disparar (tokens de contexto são caros).
    • Desvantagem: Perde o Cursor Tab (auto-complete nativo).

2. Financeiro e Repasse ao Cliente

Custo estimado do Pro em Reais: R$ 110,00 a R$ 120,00 (considerando dólar a R$ 5,20 + IOF de 4,38% + Spread bancário).

Estratégia de Repasse: Embutir como "Custo de Infraestrutura" ou "Taxa de Produtividade". Se o Cursor economizar 2-3 horas de trabalho no mês, ele já está pago.

3. Infraestrutura: Saindo do PythonAnywhere

Objetivo: Reduzir custo fixo e aumentar performance.

Opção Custo Médio Vantagem Desvantagem
PythonAnywhere (Pago) US$ 5 - 50+ Gerenciado, fácil configuração. Caro conforme escala, rígido, sem VPN real.
VPS (Hetzner / DigitalOcean) US$ 4 - 6 Muito barato, poder total, roda tudo (Docker, VPN, etc). Exige configuração manual (Linux).
Oracle Cloud (Free Tier) R$ 0,00 Melhor hardware grátis. Difícil de conseguir conta.

A Solução VPN + SSH:

  • Tailscale: Usar como VPN gratuita para criar rede privada entre seu PC e a VPS.
  • Segurança: Fechar as portas do servidor e acessar apenas via IP do Tailscale.
  • Cursor Remote SSH: Conectar o Cursor diretamente na VPS via SSH (IP da VPN) para editar código em tempo real.

4. Estratégia de Migração: Web2py -> Django

Problema atual: Web2py está lento e o ecossistema é legado.

Passos sugeridos:

  1. Mapeamento de Banco: Usar o comando python manage.py inspectdb no Django para gerar os modelos baseados no banco atual do Web2py.
  2. Padrão Estrangulador: Migrar por módulos. Manter o Web2py rodando e subir partes novas em Django no mesmo servidor.
